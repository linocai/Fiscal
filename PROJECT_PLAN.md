## 1. 项目定位

Fiscal 是一款面向个人使用的高质量财务管理 App，覆盖 iOS 与 macOS。

它不是功能极少的轻量记账工具，也不继续扩张为通用个人财务 ERP。产品聚焦于一条完整、可靠的主线：

> 通过手工或 AI/OCR 快速录入财务记录，让账户、信用负债、报销、现金流和报表始终保持一致，并允许用户完整编辑和纠正任何记录。

新项目不以迁移旧项目代码为目标。旧项目主要作为业务规则、历史决策和踩坑记录的参考来源。

### 1.1 核心目标

1. 快速、准确地记录每一笔个人财务活动。
2. 清楚回答钱在哪里、花在何处、未来要还多少。
3. 以用户维护的分类体系为报表基础。
4. 支持信用卡账期、还款、分期和报销等真实使用场景。
5. 保留 Back Tap、快捷指令、OCR 与 AI 串联的免打开 App 记账体验。
6. iOS 和 macOS 功能一致、交互分别优化，业务逻辑不重复实现。
7. 所有正式记录都有明确、安全、可追溯的编辑能力。

### 1.2 非目标

- 不做面向公众的多用户 SaaS。
- 不做 Apple 登录、用户注册、用户资料或多用户权限体系。
- 不做通用企业财务、会计或税务系统。
- 不做投资持仓、行情和每日盈亏管理。
- 不做泛化 AI 财务聊天、AI memo 或与记账无关的 AI 功能。
- 不为展示效果堆叠无业务价值的仪表盘和图表。
- 不在核心账本稳定前扩张大量系统集成。

### 1.3 前端视觉合同

`design_handoff_fiscal_app/` 中的原型与交付说明是 Fiscal 的正式视觉方向，不是最终功能清单，也不是可直接复制的生产代码。

施工约束：

- 最终 iOS/macOS 成品必须保持原型所建立的整体观感、布局语言、信息密度、语义色、字体层级、圆角、材质与交互质感。
- 原型未覆盖的功能可以继续增加，但新增页面必须从同一套设计系统自然生长，不能自行形成另一种视觉风格。
- 原型明确列出的一级导航、`更多`入口和设置分组属于目标信息架构；分 Phase 施工只影响何时接通，不得让尚未施工的目标入口在权威计划中无声消失。确需合并、移动或删除时，必须由用户确认并同步修改本文件。
- 生产前端使用原生 SwiftUI 重建，不移植原型 HTML、内联 SVG 或 `support.js`。
- iOS 与 macOS 共享术语、设计 token、业务规则和基础组件，但布局分别按移动快速操作与桌面高密度整理优化，不做简单缩放。
- 对原型未定义、且视觉判断较大的新页面，先做可运行的静态/样例数据界面并截图确认，再接入完整业务。
- 每个 Phase 都必须进行 iOS 与 macOS 的实际运行和截图验收；“能编译”不等于视觉完成。
- 视觉验收必须覆盖主页面之外的新建、编辑、预览与确认界面；裸 `Form`、默认工具栏和无设计处理的系统按钮不得作为最终成品交付，除非视觉原型明确要求。
- 当前原型是视觉下限。允许继续提高可访问性、数据密度、复杂状态和功能完整度，但不得降低成品质感。

## 2. 产品原则

### 2.1 统一账本

账户余额、信用负债、还款、分期、报销和报表必须建立在同一套正式记录之上。不得由不同模块各自维护一套金额真相。

### 2.2 业务语义与分类分离

- 餐饮、交通、购物等是用户分类。
- 信用卡消费、还款、转账是资金行为。
- 可报销、已报销是业务属性或关联状态。
- 分期是付款安排。

这些概念不得混入同一棵分类树。

### 2.3 AI 是录入层，不是另一套账本

AI 负责从文本或 OCR 内容生成记账提案。提案执行后必须成为普通正式记录，拥有与手工记录相同的查看、编辑、撤销和报表行为。

### 2.4 精简来自边界和一致性

精简不等于牺牲核心能力，而是：减少旁支模块、重复状态、重复界面、特殊记录类型和不必要的基础设施。

### 2.5 报表必须口径明确

消费、现金流和负债是三种不同视角。界面必须明确当前口径，不能将它们混成一个含义不清的数字。

## 3. 已确认功能边界

### 3.1 保留

#### 账户与流水

- 现金、储蓄卡、信用卡等账户。
- 收入、支出、转账、信用消费、还款等财务记录。
- 流水查询、筛选、搜索和完整编辑。
- 账户余额与信用负债自动更新。
- 账户归档；历史记录不得因归档丢失。

#### 信用卡

- 账单日、还款日和信用额度。
- 账期归属与账单汇总。
- 全额或部分还款。
- 分期计划、期数、手续费和每期状态。
- 当前应还、未来应还和还款进度。

#### 报销

- 一笔或多笔垫付款关联报销单。
- 一张报销单可包含多个报销付款人或付款主体，并分别记录其应付与到账情况。
- 预计报销、部分到账、全部到账等状态。
- 报销回款关联实际资金记录。
- 同时提供原始消费和个人净支出视角。

#### 现金流与报表

- 报表前端只保留消费视角；以实际个人承担为主数字，支持分类和流水下钻。
- 现金流保留为独立的未来待办/历史页面，不再作为报表 lens。
- 信用负债和未来应还归入账户／信用账户及总览，不再作为报表 lens。
- 分类、时间、账户等基础筛选。
- 首页仅展示可信且可解释的摘要，不堆叠装饰性指标。

#### AI/OCR 自动记账

- 文本记账。
- 截图 OCR。
- AI 结构化解析。
- AI 提案审核。
- 满足安全条件时自动记账。
- 自动记账结果通知与快速撤销。
- Back Tap 与快捷指令作为完整录入链路的一部分保留。

#### 全面编辑

- 流水、信用消费、还款、分期、报销、账户和分类均须提供符合业务规则的编辑能力。
- 编辑后自动重算所有受影响的数据。
- 用户不需要理解内部的撤销、重建或差额计算机制。

#### 双平台

- iOS：快速录入、移动查看、OCR、Back Tap 和快捷指令体验优先。
- macOS：密集浏览、完整编辑、批量整理和报表分析优先。
- 两端共享领域模型、业务规则、数据访问、校验与格式化能力。

#### 设置与数据管理

- 默认账户、默认记账类型和保存后是否继续记账等本地偏好。
- AI 自动记账阈值、置信度与 OCR/快捷指令来源开关；只有对应能力完成后才允许启用。
- 账户、分类和统计口径入口，以及范围经 decision gate 确认后的基础数据导出。
- 个人 VPS 连接、同步状态、设备密钥和本地缓存的真实状态管理。
- 本产品没有用户登录体系，因此不提供“退出登录”；设备侧操作使用“移除此设备密钥”或“重新配置连接”等准确语义。

### 3.2 暂不纳入

- 投资账户持仓、行情和每日盈亏。
- 订阅管理。
- 独立的对账中心；必要的核对能力收入口账户或账期场景。
- 泛化附件管理系统。
- 泛化 AI 对话和 AI memo。
- 通用通知规则编辑器。
- macOS 菜单栏独立产品形态。
- 复杂审计日志产品界面。
- 多人协作、家庭共享和权限系统。
- 除基础导出外的复杂导入导出系统。

### 3.3 已确认的首轮决策

- **币种**：仅支持 CNY，不保留 USD、汇率或折算逻辑。
- **数据真相源**：云端优先，VPS 数据库是唯一正式真相源；客户端通过 API 读写。
- **后端**：自建后端，负责正式账本、跨端同步和 AI 调用。
- **身份**：完全取消 Apple 登录、用户注册和用户体系。VPS API 仍使用设备级访问密钥保护，但密钥不代表用户身份。
- **历史迁移**：新版本完成并稳定后，从旧 LinoFinance 选择性迁移真实数据；不要求完整照搬旧库。
- **分类**：使用一套全新的分类体系，不继承旧分类结构。
- **编辑历史**：产品不提供完整版本历史；系统只保留保障数据正确性所需的最小内部追踪信息。
- **报表周期**：默认使用自然月。
- **报销**：支持部分到账及多人/多付款主体；因系统仅支持 CNY，不存在跨币种报销。
- **AI 自动执行**：沿用旧项目现行阈值——低风险、金额不超过 1000 CNY 且所需 ID/字段完整时允许自动执行；否则进入提案确认。

### 3.4 仍待后续决定

- 附件是否只服务于报销凭证，不发展为通用附件系统。
- CSV/JSON 等数据导出和备份的最低范围。
- 云端优先模式下是否需要只读缓存、短时缓存或离线草稿；VPS 始终是正式真相源。
- 设备级访问密钥的签发、轮换和撤销方式。

## 4. 统一领域模型草案

> 本节定义概念边界，不在 v0.1 锁定数据库字段和具体框架。

### 4.1 Transaction：财务事件

所有实际发生的财务活动都由统一交易对象表达，例如收入、支出、转账、信用卡消费、还款和报销回款。

建议包含：

- 唯一 ID
- 交易类型
- 发生日期与时间
- 标题与备注
- CNY 金额
- 主分类
- 可选标签
- 业务关联信息
- 来源：手工、AI 文本、OCR 等
- 创建与更新时间

### 4.2 Posting：账户影响

一笔交易可以对一个或多个账户产生影响。

- 普通支出：一个账户减少。
- 普通收入：一个账户增加。
- 转账：来源账户减少，目标账户增加。
- 信用消费：信用账户负债增加。
- 信用卡还款：付款账户减少，信用账户负债减少。

用户界面不暴露会计术语，但内部用统一账户影响避免重复计数。

### 4.3 CreditCycle：信用账期

- 属于一个信用账户。
- 保存账期起止、账单日、还款日和状态。
- 聚合关联交易，不另造一套消费真相。
- 账期金额应由关联交易和还款计算得出。

### 4.4 InstallmentPlan：分期计划

- 保存总额、期数、手续费、起始日期与信用账户。
- 关联或生成各期记录。
- 编辑计划时必须明确如何处理已发生与未发生的期次。

### 4.5 ReimbursementClaim：报销单

- 关联一笔或多笔可报销支出。
- 关联零笔或多笔报销回款。
- 包含一个或多个报销付款人/付款主体，以及各主体应付金额。
- 同一主体或不同主体均可分多次到账。
- 状态和金额由支出、各主体申报金额与到账记录共同计算。
- 不复制原始支出金额作为另一份账本真相。

### 4.6 AIProposal：AI 提案

- 保存原始输入、OCR 文本、结构化动作、置信度和状态。
- 未执行前不影响正式账本。
- 执行时调用与手工录入相同的业务服务。
- 执行完成后仅保留来源和追踪关系，不形成特殊类型流水。

## 5. 分类系统

分类系统是报表正确性的基础能力，不是简单的设置页面。

### 5.1 分类规则

- 每条正式收入或支出必须有一个主分类。
- 分类最多两级，避免无限层级。
- 收入分类和支出分类明确区分。
- 标签是可选的辅助维度，不能代替主分类。
- 转账、还款、分期、报销状态不能作为消费分类。

### 5.2 分类配置

分类由用户自行维护，系统提供以下能力：

- 名称、父分类、方向、图标、颜色和排序。
- 启用、归档。
- 是否进入日常消费汇总等报表属性。
- AI 识别别名。
- AI 匹配示例。
- 查看该分类下的流水。

### 5.3 分类编辑语义

- 重命名：历史记录自动显示新名称。
- 移动：允许调整父分类，并反映在新的报表结构中。
- 合并：将旧分类的流水迁入目标分类。
- 拆分：通过筛选和批量编辑重新分类。
- 归档：停止新选择，保留历史。
- 已被流水使用的分类不得直接物理删除。

### 5.4 待归类机制

- AI 或导入无法可靠判断时，记录进入“待归类”收件箱。
- “待归类”是数据质量状态，不是正常消费分类。
- 首页和报表应明确显示待归类数量及其对统计的影响。

### 5.5 AI 分类约束

- AI 只能返回现有分类的稳定 ID，不得自行创造分类名称。
- 提示上下文包含有效分类、别名和少量用户示例。
- AI 输出包含分类 ID、置信度及简短匹配依据。
- 分类置信度不足时不得静默自动入账。
- 用户修正分类的结果可用于后续规则或示例，但不得未经确认自动改变分类体系。

## 6. 报表口径

### 6.1 消费视角

回答：“我把钱花在了哪里？”

- 以消费实际发生日期统计。
- 信用卡消费计入消费。
- 信用卡还款不重复计入消费。
- 账户转账不计入收入或支出。
- 同时支持原始支出和扣除报销后的个人净支出。
- 主要按时间、主分类、二级分类和账户分析。

### 6.2 现金流（独立页面，不是报表）

回答：“钱在什么时候实际进出账户？”

- 展示现金与银行类账户的实际流入流出。
- 信用卡还款在现金流中出现。
- 转账单独展示或按规则排除在净现金流之外。
- 不用现金流替代消费统计，且不在「报表」内提供切换入口。

### 6.3 信用负债（账户／信用账户）

回答：“现在和未来需要还多少钱？”

- 当前信用负债。
- 各账期应还和到期时间。
- 已还、未还和逾期状态。
- 分期未来各期金额与手续费。
- 总览只展示未来 30 天内按信用账户与还款日聚合的应还；点击直接进入对应信用账户，不经过负债报表。

### 6.4 报表质量门槛

- 所有指标必须能追溯到构成它的流水。
- 分类和金额点击后能查看明细；图表不是报表完成条件。
- 明确展示时间范围和统计口径；所有金额均为 CNY。
- 分类不完整时必须提示，不伪装成精确结果。
- 同一个指标在首页、报表和账户详情中使用同一计算服务。

## 7. AI/OCR 记账链路

### 7.1 完整入口

```text
Back Tap
  → 快捷指令
  → 获取或传入截图
  → 端侧 OCR
  → AI 结构化解析
  → 本地业务校验
  → 自动执行或进入提案确认
  → 结果通知
  → 必要时快速撤销
```

同时支持直接输入文本触发同一解析与提案流程。

### 7.2 AI 输出要求

AI 输出结构化数据，不直接写数据库。至少包含：

- 交易类型
- CNY 金额
- 日期
- 来源/目标账户候选
- 分类 ID 候选
- 商户或标题
- 是否可能报销
- 字段级置信度
- 无法确定的字段

### 7.3 自动执行边界

只有满足明确安全条件的提案才能自动记账，例如：

- 金额、账户和分类均完整，金额币种固定为 CNY。
- 账户和分类 ID 确实存在且可用。
- 仅限旧项目现行规则定义的低风险记账操作，且金额不超过 1000 CNY。
- 置信度达到阈值。
- 通过幂等检查，避免快捷指令或系统重试造成重复记账。

不满足条件时必须进入提案确认，不得猜测填充后静默落账。

### 7.4 撤销

- 自动记账成功后提供结果通知。
- 支持从通知快速撤销最近一次 AI 操作。
- 撤销走正式账本的统一修改机制，不直接删除局部数据。

## 8. 全面编辑机制

### 8.1 基本流程

```text
读取完整业务对象
  → 用户修改
  → 校验新状态
  → 计算修改前后的财务影响
  → 撤销旧影响
  → 应用新影响
  → 重算账户、账期、分期、报销和报表
  → 保存变更记录
```

### 8.2 编辑范围

- 流水：金额、日期、账户、分类、标题、备注和业务属性；币种固定为 CNY，不作为可编辑字段。
- 信用消费：金额、日期、信用账户和账期归属。
- 还款：付款账户、信用账户、金额和日期。
- 分期：总额、期数、手续费、起始期和相关账户。
- 报销：关联支出、付款主体、各主体申报金额、到账记录和状态。
- 分类：重命名、移动、合并、拆分和归档。
- 账户：允许字段修改与归档。

### 8.3 复杂编辑规则

- 修改已经结清账期中的流水时，必须提示影响范围并重新计算。
- 修改已有还款关联的账期时，不能产生负数或重复还款状态。
- 修改已有部分到账或多主体付款的报销单时，必须保持各主体应付、已到账、支出与回款可追溯。
- 修改已经发生部分期次的分期计划时，必须区分已发生和未来期次。
- UI 对用户呈现业务影响，不暴露数据库层操作。

### 8.4 最小变更追踪

产品不提供完整版本历史页面，也不以保存每次字段变化为目标。具体采用“原记录作废并重建”或“原记录更新加最小内部追踪”，留到技术设计阶段决定。无论采用哪种方式，都必须满足：

- 用户看到的是一条连续、可理解的记录。
- 修改前后可追溯。
- 内部追踪仅保留保证关联修正、幂等和问题排查所需的信息。
- 关联对象不会指向失效记录。
- 报表不会重复计数。

## 11. 技术方向与已定约束

以下作为 P1 的默认技术基线；若施工期出现有证据的重大阻碍，必须先回写本文件说明原因，再变更选型：

- 前端采用原生 SwiftUI 多平台工程，iOS/macOS 共用领域层、数据层、Feature Model、设计 token 与基础组件，平台分别实现必要的壳层和布局。
- 最低部署版本固定为 iOS 26 与 macOS 26。本项目仅供用户本人使用，不为旧系统做向下兼容、availability 分支或 back-deployment 包袱。
- Swift 使用施工时当前 Xcode 正式版所提供的最新稳定工具链与语言模式；当前已验证基线为 Xcode 26.6、Swift 6.3.3，并启用 Swift 6 严格并发检查。
- 客户端优先采用当前原生能力，包括 SwiftUI App 生命周期、Observation、Swift Concurrency（async/await、Task、actor）、Swift Testing 和现代 Foundation API。除非有明确系统限制，不回退到 Combine/ObservableObject、回调式异步或仅为旧系统存在的实现方式。
- 后端采用 FastAPI、SQLAlchemy、Alembic 与 PostgreSQL。
- 数据采用云端优先架构，部署在个人 VPS；服务端数据库是唯一正式真相源。
- 自建后端统一承载账本 API、跨端数据同步和 AI Provider 调用。
- 客户端不包含 Apple 登录或用户体系；API 通过 HTTPS 和存放于 Keychain 的设备级访问密钥保护。
- 业务层保持平台无关，避免 iOS/macOS 各自维护一份状态模型。
- 财务金额使用 `Decimal` 或明确的最小货币单位表示，禁止二进制浮点数。
- 全系统仅使用 CNY，不引入汇率表、折算金额或多币种展示。
- 正式记录的写入、编辑和撤销必须经过统一业务服务。
- 报表由可测试的纯计算或查询层生成，View 不自行拼装口径。
- AI Provider 与账本写入隔离，所有 AI 输出先进入结构化校验。
- 不直接继承旧后端实现；先根据新领域模型重新设计 API 和数据表。
- 原型仅作为视觉与交互参考，生产工程不引入其 HTML runtime。

## 12. P1–P18 施工计划

### 12.1 固定施工循环

每个 Phase 都按同一顺序完成：

1. 明确本期业务规则、状态机、异常场景与验收样例。
2. 设计数据库迁移和 API 契约。
3. 实现后端、自动化测试和稳定的错误语义。
4. 实现 Swift 共享 DTO、Repository、领域服务和 Feature Model。
5. 实现 iOS 与 macOS 对应界面，不把任一平台长期留到后补。
6. 使用真实 API 数据联调，不以静态 mock 作为本期完成条件。
7. 验证创建、查看、编辑、删除/撤销、空、加载、错误、禁用与长内容状态。
8. 分别运行 iOS/macOS 并截图，与前端视觉合同对照修正。
9. 执行代码审查、测试、迁移检查和本地 Git 提交。
10. 到达关键产品节点时交给用户目视/真机验收，再进入下一期。

每个 Phase 必须交付一个可独立运行的纵向切片。禁止出现后端连续推进多期、前端最后一次性补齐的施工方式。

### P1：工程地基与视觉基线

目标：建立可以长期施工的正式工程，并将现有原型转化为原生前端基线。

后端：

- FastAPI、SQLAlchemy、Alembic、PostgreSQL 工程骨架。
- 统一配置、错误结构、日志、日期、时区和金额规范。
- 健康检查、测试框架和本地开发数据库。
- VPS 开发/预发布部署骨架。
- HTTPS 与设备级 API Token 骨架；不建立 Apple 登录、用户表或多用户权限。

前端：

- 最低部署版本为 iOS 26/macOS 26 的 SwiftUI 多平台工程，使用当前最新稳定 Swift 语言模式与原生 API，不建设旧系统兼容层。
- 共享 API Client、Repository、错误处理与 Keychain 基建。
- 从视觉合同提炼颜色、字体、间距、圆角、阴影、语义色和基础组件。
- 建立 iOS Tab Shell 与 macOS Sidebar Shell。
- 使用样例数据高保真实现双端总览视觉基线。

验收：双端构建运行、连接本地/VPS 健康接口、总览观感达到原型水准。本期结束由用户做第一次视觉确认。

### P2：账户与分类

目标：建立所有正式流水依赖的基础资料。

- P2 开工先修复 iOS Shell：当前用隐藏系统 `TabView` 再叠加自定义玻璃 Tab Bar，在 iOS 26 仍会残留系统 Tab Bar，形成底部双导航栏。改为单一导航源（自定义 Tab Bar + 显式内容切换），不得继续依赖“隐藏原生 Tab Bar”的叠层方案。
- 现金、储蓄卡、信用卡账户；期初余额、排序、编辑和归档。
- 信用额度、账单日、还款日等信用账户字段。
- 收入/支出分类，最多两级。
- 分类图标、颜色、排序、AI 别名与识别示例。
- 分类新建、编辑、移动、合并、拆分辅助、归档和使用情况查看。
- “待归类”作为数据质量状态，不作为普通消费分类。
- iOS 账户/分类设置与 macOS 账户卡片/高密度分类管理。

验收：iOS 全部 P2 页面始终只出现一套底部导航栏；所有基础资料完整可编辑；已使用的账户和分类不能被破坏性删除；分类结构可直接支撑后续报表。

### P3：统一账本与完整编辑

目标：交付收入、支出和转账的正式手工记账闭环。

- 统一 Transaction + Posting 账户影响模型。
- CNY 单币种；收入、支出、转账。
- 创建、查看、编辑、删除/撤销与最小内部追踪。
- 修改后自动重算账户余额。
- 服务端幂等键，防止重复提交。
- 转账必须同时选择来源账户与目标账户，且不计入收入/支出。
- iOS“记一笔”、流水分组/筛选/搜索/编辑。
- macOS 流水表格、筛选、选择与右侧 Inspector。

验收：任意流水编辑后余额与分类汇总正确；删除、撤销和重复提交不会污染账本。P3 后 Fiscal 已具备可日常使用的基础记账能力，由用户验收录入与编辑手感。

### P4：信用卡账期与还款

目标：在统一账本上完成信用负债闭环。

- 信用消费自动归属账期。
- 账期起止、账单日、还款日、当前已用、本期应还和剩余未还。
- 全额/部分还款，明确付款账户与信用账户。
- 已结清、未结清、逾期状态。
- 信用消费、还款和允许字段的完整编辑与重算。
- 信用消费计入消费；还款不重复计入消费，但计入现金流并减少负债。
- iOS 信用账期详情/还款表单，macOS 信用账户详情/账期管理。

验收：账户、账期、消费、还款和负债始终一致。

### P5：分期

目标：将分期纳入统一信用账本，不创建第二套金额真相。

- 分期总额、期数、手续费、起始账期和各期应付日期。
- 已发生与未来期次，与信用消费、账期和还款关联。
- 编辑未来期次；保护已发生或已还款期次。
- 提前结清与取消未发生期次的明确业务规则。
- iOS 分期详情/编辑，macOS 分期计划/期次列表/未来负债。

验收：分期消费只统计一次；每期与未来计划负债可解释，且不把尚未发生的还款伪装成现金流；编辑计划不破坏历史账期。

### P6：多人/多主体报销

目标：完成垫付、申报、多主体付款和分次回款闭环。

- 一张报销单关联一笔或多笔垫付款。
- 多个付款人/付款主体及各主体应付金额。
- 同一或不同主体分多次到账。
- 待提交、待付款、部分到账、全部到账等状态。
- 回款作为正式资金记录。
- 关联支出、主体、应付和到账记录的完整编辑。
- 消费报表区分原始消费、预计可报销与个人最终承担；现金流只反映真实到账。
- iOS 报销查看/快速登记到账，macOS 报销管理/主体拆分/完整编辑。

验收：各主体应付、已付、未付与总额守恒；回款不被重复计作普通收入。

### P7：总览、现金流与三类报表

目标：在业务口径稳定后，正式解决旧项目报表失真和信息不足的问题。

- 消费报表：本月消费、个人净支出、一级/二级分类、日/月趋势和明细下钻。
- 现金流：自然月实际流入/流出/净额、未来 30 天事件与按账户分析。
- 负债：当前信用负债、各账期、分期未来应还、到期与逾期。
- 分类缺失和待归类对统计的影响必须显式呈现。
- 总览只提取少量可信摘要，每项指标均可下钻。
- iOS 保持重点摘要与列表：现金流使用独立页面，报表只包含消费和负债，不重复现金流，也不展示图表；macOS 提供高密度图表分析、周期选择和更完整明细。

验收：三种口径不互相污染；相同指标在总览、账户和报表中使用同一计算服务。P7 结束由用户重点验收报表是否真正可信、有解释力。

### P8：AI 记账后端与文本提案

目标：完成 AI 记账大脑、安全边界和文本入口。

- AI Provider 抽象与服务端密钥管理。
- 文本输入解析为结构化 AIProposal。
- 账户、分类、日期和金额匹配；AI 只能返回现有稳定 ID。
- 字段级置信度、内容指纹、幂等和提案状态机。
- 提案执行、编辑、忽略和撤销。
- AI 执行调用与手工录入相同的正式账本服务。
- 自动执行沿用旧规则：仅低风险、金额不超过 1000 CNY、必要 ID/字段完整且达到置信度。
- iOS/macOS AI 待确认与提案编辑；执行后成为可正常编辑的普通流水。
- 恢复完整产品入口：iOS `更多 → AI 待确认`、总览待确认状态，以及 macOS `AI 待确认`侧栏；不得因为此前 Phase 尚未实现而继续缺席。
- iOS `更多`中的 `其他设置 · P11`占位必须在本期改为正式 `设置`目的页；P8 先交付 AI 自动记账分组，后续分组按 P9–P11 接通。
- 接通设置页中的 AI 自动记账开关、金额上限和最低置信度；服务端安全规则仍是最终约束，客户端设置不能放宽后端边界。

验收：AI 永远不能绕过账本校验，重试不会造成重复记账。

### P9：OCR、快捷指令、Back Tap 与通知撤销

目标：完成免打开 App 的实际日常记账闭环。

```text
Back Tap
  → 快捷指令
  → 获取或传入截图
  → 端侧 Vision OCR
  → 后端 AI 解析
  → 安全判断
  → 自动记账或待确认
  → 本地通知/播报
  → 快速撤销
```

- App Intent 文本记账。
- 截图输入、相册最新截图读取和 Vision OCR。
- 快捷指令参数接线与 Back Tap 使用路径。
- 接通设置页中的 OCR 与快捷指令文本来源开关，并展示真实授权/可用状态。
- 自动记账结果播报、本地通知和通知内撤销。
- 机械重试幂等、网络失败和 AI 未配置的清晰中文反馈。

验收：除编译与模拟器门禁外，必须由用户进行 Siri、Back Tap、相册、通知和撤销真机验收。

### P10：双端完整体验与复杂编辑

目标：将所有“能用”提升到稳定成品。

- 补齐所有业务对象的编辑表单和修改影响确认。
- 待归类收件箱、批量重新分类、全局搜索与高级筛选。
- iOS 导航、复杂表单、滚动、键盘和安全区优化。
- macOS Inspector、批量整理、键盘快捷键和窗口适配。
- 空、加载、错误、离线、禁用、超长内容和大数据量状态。
- Dynamic Type、VoiceOver、Reduce Motion；正式决定并实现深色模式范围。
- 分页、并发刷新、重复请求、短时缓存和性能治理。
- 完成 iOS `更多 → 设置`与 macOS 设置页的正式壳层：记账偏好、分类/统计入口、数据与缓存；默认账户/类型和保存后停留必须真正影响记账流程。
- 对 CSV/JSON/PDF 基础导出范围执行 decision gate；只实现确认后的最小范围，不把复杂导入导出系统带回项目。

验收：iOS/macOS 分别完成真实视觉与交互巡检。本期不是第一次做双端，而是集中偿还成品质感与复杂状态债务。

### P11：VPS 生产化与数据安全

目标：让系统能够安全、可恢复地长期运行。

- PostgreSQL 生产数据库、HTTPS、反向代理和服务进程。
- 设备 Token 哈希存储、签发、轮换和撤销。
- 生产配置与密钥隔离，AI Key 仅存服务端。
- 数据库自动备份、恢复演练、健康检查、错误日志、磁盘监控和 API 限流。
- Alembic 正式迁移流程与故障恢复策略。
- 明确真实可实现的传输/存储加密边界；不提供与服务端计算、AI 处理相冲突的虚假“端到端加密”开关。
- 接通设置页中的 VPS/同步、设备密钥、备份与安全状态；因为项目没有用户体系，不实现“退出登录”，改用准确的设备密钥移除/重新配置操作。

验收：备份可恢复、密钥可轮换、服务重启不丢数据、生产环境不暴露开发配置。

### P12：选择性迁移、对账与 v1.0 发布

目标：迁移需要的旧数据并正式切换到 Fiscal。

### P18：v1.2.3 总览、信用应还与消费报表收敛

**状态：已立项，未施工。** 本期只收敛现有派生读模型和双端展示；不改正式账本、信用账期、报销、现金流事项的写入语义。目标是让用户在总览清楚看到现金余额、近期流水和未来应还，并让「报表」回到一个可解释、可下钻的消费页面。

#### 已锁定目标与非目标

- 总览的最近流水由服务端统一上限为 10 条，两端完整渲染这 10 条；macOS 行高固定紧凑，不以卡片剩余高度拉伸。
- 总览把「报销待回款」卡替换为「现金余额」，数值复用现有 `account_value_minor`（现金 `cash` + 借记卡 `debit` 的派生余额），点击进入账户。报销模块、回款、`reimbursement_outstanding_minor` 和其后端兼容字段都不删除。
- macOS 总览的「消费口径」改为一个主数字「本月自己承担」=`personal_realized_minor`；辅助文案说明它已扣商家退款和已到账报销，未到账报销仍计入本人承担。iOS 使用同一术语与同一值。
- 总览新增「未来 30 天信用应还」：仅显示仍有余额的信用账期，按 **信用账户 ID + 还款日** 聚合，日期窗口沿用现行 `[today, today + 30)` 的 30 个自然日定义。内联最多 4 条；恰好当前的 4 条必须全见，超过 4 条显示「查看全部」并在独立列表展示全部聚合行。任一行都直达相应信用账户，绝不跳负债报表。
- 报表双端只保留消费页：删除消费／现金流／负债切换、Mac 现金流和负债报表视图及其图表。保留 iOS 独立「现金流」Tab 与 macOS「现金流」侧栏；信用负债继续在账户／信用账户查看。后端旧报表端点可继续兼容，不以删接口或删数据迁移作为目标。
- 消费报表的主数字固定为「实际个人承担」=`personal_realized_minor`。一级分类和二级分类均按该金额降序，金额相同按现有稳定分类顺序打破平局；每行数字优先展示金额、占总实际个人承担的占比、流水数，并保留原有分类下钻。若总额小于等于零，占比显示 `—`，不制造误导百分比；待归类继续作为数据质量提示，不伪装成普通分类。
- 本期删除 Mac 消费趋势图和仅为现金流／负债报表服务的图形代码；iOS 仍无图表。保留后端 `trend` 字段以保证旧客户端兼容。

明确不做：不改报销状态或金额计算；不改信用账期/还款/分期规则；不删除独立现金流产品；不删除 `GET /reports/cash-flow`、`GET /reports/debt`、现金流 drill-down 或其 DTO；不做账户余额历史曲线、跨端离线同步或新的数据库表。

#### 当前事实、风险与技术决定

- 当前 `GET /reports/overview` 已计算 `account_value_minor`（`cash` + `debit` 的期初余额加 posting 影响），因此现金余额只需改展示和导航，无数据库迁移、无重复计算。
- 现状总览只截取 5 条流水；iOS 已全量显示响应，macOS 的 `.frame(maxHeight: .infinity)` 会把行拉高。服务端改为 10 条后 macOS 统一使用约 32 pt 紧凑行及自然滚动，不能裁切或撑满整张卡。
- P17 的未来现金流投影已经按账户与还款日分组，但 `ReportingService._forecast` 仍逐账期生成 `ForecastEvent`。P18 以报告服务为唯一总览聚合点，不从 SwiftUI 合并金额，避免重复或与信用账户不一致。
- 现有两个独立 `ReportingModel`（总览、报表）各自 `loadAll()`；它们会同时请求 cash-flow/debt，且报表把所选月的 `dateTo` 传给 debt `as_of`，在当月会错误使用月末的未来日期。保留两个模型以防总览重置报表月份，但改为按用途加载，并通过共享失效协调器同时刷新；前端不再请求 debt/cash-flow 报表端点，彻底消除该 `as_of` 错用。
- `v1.2.1` 已发布且 tag 已存在，`MARKETING_VERSION=1.2.1`、`CURRENT_PROJECT_VERSION=11`，仓库没有 `v1.2.2` 或 Build 12 记录。本期固定为 **v1.2.3 (Build 12)**；Build 号按安装顺序递增，不因跳过营销版本而跳号。

#### 接口、共享模型与兼容契约

1. `GET /api/v1/reports/overview` 保持现有字段和 `forecast`，新增非破坏性 `credit_due_events`。每个元素至少为：`account_id`、`account_name`、`due_date`、`remaining_minor`、稳定排序用的 `cycle_ids`（或等价的可追溯成员 ID 集合）。服务端仅对未结清、窗口内账期聚合；分组金额必须等于成员 `remaining_minor` 之和，排序为还款日、账户名、账户 ID。旧客户端忽略该新字段。
2. 现有 `account_value_minor` 是现金余额唯一来源，客户端不得重新遍历账户或 posting 相加。`reimbursement_outstanding_minor`、`forecast`、`/reports/cash-flow`、`/reports/debt`、`/reports/drill-down?lens=cash_flow` 保持后端兼容；P18 新客户端不再把它们作为报表 UI 依赖。
3. Swift `OverviewReport` 增加严格解码的 `creditDueEvents`；新增 `OverviewCreditDueEvent`。`ReportingRepository`/模型拆分为「总览加载」和「消费报表加载 + spending drill-down」，不保留前端 debt/cash-flow `loadAll` 路径。任何仍需 debt 的兼容调用必须以 Asia/Shanghai 的实际当天为 `as_of`，不得传自然月最后一天。
4. 总览导航契约：现金余额 → 账户入口；信用应还行 → 带 `accountID` 的信用账户详情。iOS 在 `IOSMoreDestination` 增加可哈希的信用账户深链目的地，macOS 向 `MacAccountsCreditScreen` 传入/更新初选信用账户 ID；两个详情页仍调用既有 `CreditModel.loadAccount(accountID)`，不传金额或账期作为不可信 UI 参数。

## 13. 质量门槛

### 13.1 财务正确性

- 每笔资金影响可解释、可追溯。
- 转账、还款和报销不得造成收入/支出重复统计。
- 编辑和撤销后所有派生数据可稳定重算。
- 金额、日期和时区有统一规范；币种固定为 CNY。

### 13.2 产品质量

- 重要指标可下钻到流水明细。
- 分类缺失和数据异常明确可见。
- 所有写入都有成功、失败和重试反馈。
- 所有业务对象都有明确的创建、编辑、归档或撤销路径。

### 13.3 界面质量

- 财务界面安静、紧凑、可扫描，不采用营销页式布局。
- iOS 与 macOS 分别验证，不以“能编译”代替实际交互验收。
- 覆盖空、加载、错误、禁用和长内容状态。
- 图表只在确实帮助理解数据时出现。

### 13.4 工程质量

- 核心业务规则有自动化测试。
- 报表口径通过固定样例验证。
- AI 永远不能绕过正式业务校验。
- 平台 UI 不承载不可复用的财务计算逻辑。

## 15. 已拍板决策记录（2026-07-13）

1. **币种**：CNY 单币种。
2. **数据真相源**：云端优先，部署在个人 VPS。
3. **后端**：AI 调用、账本 API 和跨端同步均使用自建后端。
4. **身份**：完全取消 Apple 登录和用户体系；保留非用户身份的设备级 API 保护。
5. **历史迁移**：需要，但在新版本完成后选择性迁移。
6. **分类初始方案**：使用全新分类体系。
7. **编辑历史**：不提供完整版本历史。
8. **报表周期**：默认自然月。
9. **报销口径**：支持部分到账和多人/多付款主体，币种固定 CNY。
10. **AI 自动执行阈值**：照抄旧项目现行规则，即低风险、金额不超过 1000 CNY 且必要 ID/字段完整时可自动执行。
11. **前端方向**：`design_handoff_fiscal_app/` 是正式视觉合同；允许继续增加功能，但最终原生成品必须保持该原型建立的观感和跨端设计语言。
12. **总体施工**：前后端由同一施工主线承建，按 P1–P12 纵向切片推进；每期包含后端、共享层、iOS、macOS、联调、截图和验收，不采用前后端长期分离施工。
13. **Apple 平台与 Swift 基线**：最低部署版本固定为 iOS 26 与 macOS 26；使用施工时最新正式 Xcode/Swift 稳定工具链和现代原生 API，不承担旧系统兼容成本。当前验证工具链为 Xcode 26.6、Swift 6.3.3。
14. **iOS `更多`与设置完整性**：目标入口固定为账户、信用账期与分期、报销、消费/负债报表、AI 待确认、分类设置和设置；分 Phase 接通不得演变为删除入口。设置能力分属 P8–P11，不整体归入 P11，也不提供“退出登录”或虚假“端到端加密”开关。

尚未完全拍板的附件范围、离线行为、设备密钥生命周期和迁移选择规则，分别在首次产生真实依赖的 Phase 内形成 decision gate，不阻塞 P1 开工。
