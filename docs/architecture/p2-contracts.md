# Fiscal P2 Accounts and Categories Contract

Date: 2026-07-15

This document freezes P2 boundaries before implementation. `PROJECT_PLAN.md` remains authoritative. P2 creates only account and category master data; it must not introduce transactions, postings, statement cycles, repayments, installments, reports, reimbursements, or AI persistence.

## Shared invariants

- Single user, CNY only, authenticated by the existing device Bearer token.
- Identifiers are UUIDs generated by the server.
- Monetary values cross the API as signed integer CNY minor units. JSON floating-point money is forbidden.
- All timestamps are UTC ISO 8601. Display dates use `Asia/Shanghai` where needed.
- Lists are stable by `sort_order`, then creation time and ID.
- Mutable resources carry an integer `version`. Direct resource mutations include the expected version; stale writes return `409 resource_version_conflict`. Complete-set ordering is the deliberate exception and is validated atomically against the current active sibling set.
- Archive is reversible. Destructive deletion is allowed only while `usage_count == 0` and no dependent master data exists.
- Because this is a single-user, low-frequency master-data surface, every P2 mutation is serialized inside PostgreSQL with one transaction-scoped advisory lock before reading validation state. This closes parent/archive, merge, and complete-set ordering races without adding distributed locking machinery.

## Accounts

Kinds:

- `cash`: physical cash or cash-like balance.
- `debit`: bank/debit asset account.
- `credit`: credit-card liability account. P2 stores configuration and an opening amount owed; formal statement cycles begin in P4.

Fields:

- `id`, `name`, `kind`, `institution`, `last_four`
- `opening_balance_minor`
- credit-only `credit_limit_minor`, `statement_day`, `due_day`
- `sort_order`, `archived_at`, `usage_count`, `version`, `created_at`, `updated_at`

Rules:

- Name is trimmed and 1–80 characters. Active account names are unique case-insensitively.
- `last_four`, when present, is exactly four ASCII digits.
- Cash/debit accounts reject credit-only fields. Credit accounts require a positive limit and statement/due days from 1 through 28.
- A credit opening amount is the amount owed and must be non-negative. Cash/debit opening balances may be signed.
- Credit amount owed must not exceed the configured limit at creation or update.
- Archiving preserves the account. Restore rechecks active-name uniqueness.
- Delete returns `409 account_in_use` once referenced; P2 exposes `usage_count` as zero until P3 introduces ledger references.

Routes under `/api/v1`, all protected:

- `GET /accounts?include_archived=false`
- `POST /accounts` → `201`
- `GET /accounts/{id}`
- `PATCH /accounts/{id}` with `expected_version` in the JSON body
- `DELETE /accounts/{id}?expected_version=` → `204` only when safe
- `POST /accounts/{id}/archive` with `{ "expected_version": integer }`
- `POST /accounts/{id}/restore` with `{ "expected_version": integer }`
- `PUT /accounts/order` with `{ "ordered_ids": [UUID, ...] }` containing every active account ID exactly once

Account list/order responses are bare JSON arrays. Create/get/update/archive/restore return one account object. Delete returns an empty `204` response.

## Categories

Fields:

- `id`, `name`, `direction` (`income` or `expense`), `parent_id`
- `icon` (SF Symbol semantic name), `color_hex` (`#RRGGBB`)
- ordered unique `aliases` and `examples`
- `sort_order`, `archived_at`, `usage_count`, `version`, `created_at`, `updated_at`

Rules:

- Hierarchy is at most two levels: root and child. A child cannot be a parent.
- Parent and child directions must match. Direction cannot change while children exist.
- Active sibling names are unique case-insensitively.
- Alias/example values are trimmed, deduplicated case-insensitively, individually capped at 40 characters, and capped at 20 values per field.
- “待归类” is a ledger data-quality state, not a category row and not returned here.
- Archive is rejected while active children exist. Delete is rejected when used or when children exist.
- Merge requires distinct active categories with the same direction and matching levels: root merges into root, child into child. P2 archives the source and moves conflict-free children when both are roots; a source child whose name already exists under the target is archived so no active child can remain under an archived root. P3 must reuse this operation to reassign ledger references transactionally.
- Split assistance in P2 means atomically creating two or more children beneath an active root. It does not create or reassign transactions.

Routes under `/api/v1`, all protected:

- `GET /categories?direction=&include_archived=false` returns roots with nested children
- `POST /categories` → `201`
- `GET /categories/{id}`
- `PATCH /categories/{id}` with `expected_version` in the JSON body
- `DELETE /categories/{id}?expected_version=` → `204` only when safe
- `POST /categories/{id}/archive` with `{ "expected_version": integer }`
- `POST /categories/{id}/restore` with `{ "expected_version": integer }`
- `PUT /categories/order` with `{ "parent_id": UUID | null, "ordered_ids": [UUID, ...] }` containing one complete active sibling set for a single direction (inferred from the first ID)
- `POST /categories/{source_id}/merge` with `{ "target_id": UUID, "source_expected_version": integer, "target_expected_version": integer }`
- `POST /categories/{root_id}/split` with `{ "root_expected_version": integer, "children": [CategoryDraft, ...] }` containing two or more child drafts

Category list/order responses are bare JSON arrays; list items are roots with nested children. Create/get/update/archive/restore return one category object. Merge returns the updated target object, split returns the newly created children array, and delete returns an empty `204` response. Domain failures retain the shared `{ "error": { "code", "message", "request_id", "details" } }` envelope.

## Stable domain errors

- `account_not_found`, `category_not_found`
- `account_name_conflict`, `category_name_conflict`
- `account_in_use`, `category_in_use`, `category_has_children`
- `invalid_account_configuration`, `invalid_category_hierarchy`
- `resource_version_conflict`
- Existing `authentication_required`, `invalid_device_token`, and `validation_error` remain unchanged.

## Apple client contract

- Shared `Codable`, `Sendable` DTOs, API error decoding, repository protocols/implementations, and `@Observable @MainActor` feature models serve both apps.
- No P2 live screen may fall back to `OverviewFixtures`; loading, empty, offline, unauthorized, validation, conflict, and unexpected failure are explicit states.
- Account/category edit forms validate locally but server rules remain authoritative.
- iOS starts P2 by removing the hidden native `TabView` beneath the custom glass bar. One selection state drives explicit content switching, so exactly one bottom navigation bar exists.
- iOS exposes Accounts and Categories from More; macOS enables the Accounts sidebar destination and a dense Categories management destination.
- Archive/restore and delete confirmations explain consequences. Merge and split are visible only when their preconditions are met.
